# 01 · Architecture & Install

How this git repo turns into a working desktop, and the rules that keep it from
breaking.

## The symlink model

This repo does **not** copy files into place. It *symlinks* them. A symlink is a
filesystem pointer: `~/.config/hypr` isn't a folder, it's a signpost that says
"the real folder is at `~/dotfiles/.config/hypr`".

```
~/.config/hypr ──symlink──▶ ~/dotfiles/.config/hypr   (the real files, in git)
~/.zshrc       ──symlink──▶ ~/dotfiles/home/.zshrc
```

**Why symlinks instead of copies:** you edit a config once, in its normal location,
and the change is already "in the repo" — ready to `git commit`. No copy step, no
drift between "the version I'm running" and "the version I committed."

### Checking what something is

```bash
ls -la ~/.config/starship          # an arrow "->" means it's a symlink
readlink ~/.zshrc                  # prints the target, or nothing if not a symlink
```

⚠️ **The footgun:** if a symlink gets *replaced by a real file* (some apps and some
HyDE operations do this), your edits no longer land in the repo. They live only in
`~/.config/...` and are invisible to git. If you ever find a config that "won't
commit," check whether it's still a symlink. Re-running `install.sh` restores the
link (backing up the stray copy first).

> A real instance of this: `~/.config/kitty/kitty.conf` was found to be a plain
> file, not a symlink, even though `install.sh` creates it as one. The fix is to
> keep the dotfiles copy authoritative and re-run `install.sh` to relink.

## `install.sh`, step by step

The installer (`~/dotfiles/install.sh`) is idempotent — safe to run any number of
times. Its core is the `link()` function:

```bash
link() {
    local src="$1" dst="$2"
    # 1. If dst already points at src, do nothing.
    # 2. If dst exists (file/dir/other symlink), MOVE it to a timestamped backup
    #    at ~/.dotfiles-backup-YYYYMMDD_HHMMSS/ before linking.
    # 3. Create the parent dir, then `ln -sf src dst`.
}
```

So nothing is ever destroyed — anything in the way is backed up to
`~/.dotfiles-backup-<timestamp>/`.

What it links, in order:

1. **Hyprland** — the entire `.config/hypr` directory.
2. **HyDE** — `config.toml`, plus each theme's `.theme` files individually.
   - ⚠️ Theme *directories* are created as real dirs (not symlinked), because
     HyDE's theme scanner uses `find -type d` and skips symlinked directories.
     Only the files *inside* each theme are symlinked. Wallpaper folders are
     symlinked dirs (HyDE tolerates that for wallpapers).
3. **Waybar** — only `user-style.css` (your overrides). The generated `config.jsonc`
   and `style.css` are HyDE's, not tracked.
4. **Kitty** — `kitty.conf`.
5. **Neovim** — the entire `.config/nvim` directory.
6. **Shell + Prompt** — `~/.zshrc` and the `.config/starship` directory.
7. **Tmux** — `~/.tmux.conf`.
8. **CLI tools** — btop, fastfetch, dunst.
9. **Firefox Dev Edition** — finds your `*.dev-edition-default` profile and links
   `userChrome.css`, `userContent.css`, `user.js` into it. Also links the wallbash
   `firefox.dcol` so Firefox colors follow your theme.

After linking it prints next steps (reload Hyprland, pick a theme, install tmux
plugins, etc.).

## Tracked vs generated: the safe/unsafe map

This is the most important table in the docs. **Green = your file, edit freely.
Red = generated, your edits get overwritten.**

| Path | Status | Notes |
|---|---|---|
| `.config/hypr/userprefs.conf` | ✅ yours | Your personal Hyprland tweaks go here |
| `.config/hypr/keybindings.conf` | ✅ yours | But HyDE may offer template updates |
| `.config/hypr/windowrules.conf` | ✅ yours | Per-app opacity/float rules |
| `.config/hypr/monitors.conf` | ✅ yours | Generated by `nwg-displays`, but tracked |
| `.config/hypr/themes/` | ⛔ generated | Rewritten on every theme/wallpaper switch — **never edit** |
| `.config/kitty/kitty.conf` | ✅ yours | `include`s the generated files below |
| `.config/kitty/hyde.conf` | ⚠️ HyDE base | HyDE-provided; overridable from kitty.conf |
| `.config/kitty/theme.conf` | ⛔ generated | Wallbash colors — **never edit** |
| `.config/waybar/user-style.css` | ✅ yours | The only Waybar file you own |
| `.config/waybar/config.jsonc`, `style.css` | ⛔ generated | By `wbarconfgen` |
| `.config/starship/starship.toml` | ✅ yours | The prompt |
| `home/.zshrc`, `home/.tmux.conf` | ✅ yours | Shell + multiplexer |
| `~/.zshrc.local` | ✅ yours (untracked) | Machine-local secrets; gitignored |
| `.config/nvim/` | ✅ yours | LazyVim config |
| `.config/nvim/lazy-lock.json` | ✅ yours | Plugin lockfile — commit it to pin versions |

**Rule of thumb:** if a file has a header like *"Generated by …, do not edit"* or
lives under a `themes/` directory, it's downstream of the
[theming pipeline](02-theming-pipeline.md). Change the *input* (wallpaper/theme),
not the *output*.

## Fresh-machine bootstrap

```bash
# 1. Install Arch + HyDE first (HyDE has its own installer).
#    https://github.com/HyDE-Project/HyDE
# 2. Install the CLI tools this setup expects:
sudo pacman -S zsh tmux kitty btop fastfetch eza zoxide fzf bat fd ripgrep \
               lazygit starship neovim
#    (oh-my-zsh + lazydocker + TPM are installed separately, see below)
# 3. Clone and link:
git clone https://github.com/Imad-Oute/dotfiles ~/dotfiles
cd ~/dotfiles && chmod +x install.sh && ./install.sh
# 4. Post-install:
hyprctl reload                       # apply Hyprland config
#    Super+Shift+T  → pick a theme
#    tmux, then  prefix + I  → install tmux plugins (prefix = Ctrl+Space)
```

Wallpapers are not in the repo (binary bloat). See [`wallpapers.md`](../wallpapers.md).

## How a change flows back to GitHub

```bash
cd ~/dotfiles
# you already edited ~/.config/foo (a symlink), so the change is here:
git status
git add -A
git commit -m "tweak foo"
git push
```

Because edits land in the repo via symlinks, your normal editing *is* your
version control workflow. The only discipline required: occasionally `git status`
in `~/dotfiles` to catch drift.
